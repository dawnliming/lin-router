## 0. 执行摘要

v0.5.5 已完成连接组创建、模型获取/验证、客户端接入等上手闭环。继续在单体 `app.py` 与单文件 `config-tab.js` 上叠加功能，会持续放大以下问题：改动影响面难判断、测试定位困难、配置页的表单/运行态/DOM 更新互相干扰。

v0.5.6 不引入新用户能力，不改变路由、聚合、配置、日志或 UI 规则；它的交付价值是为后续版本提供**可定位、可测试、可小步演进的职责边界**。

```text
当前：功能能用，但职责集中、变更风险随文件增长而上升
  ↓
v0.5.6：保持行为与接口不变，建立模块边界、回归基线和迁移顺序
  ↓
后续：新功能在对应领域模块内演进，而非继续堆入 app.py / config-tab.js
```

### 0.1 本期成功标准

1. `app.py` 从“数据模型 + 持久化 + 路由运行时 + HTTP Handler + 启动”混合职责，收敛为应用装配/启动入口和薄 HTTP 路由分发；领域实现迁出到职责明确的模块。
2. `config-tab.js` 从“页面渲染 + 表单草稿 + 校验 + 状态推导 + 自动保存 + API 动作 + 运行态刷新”混合职责，收敛为页面协调入口；表单与运行态更新不再以整页重渲染相互影响。
3. 对外 API 路径、HTTP 方法、请求/响应字段、错误机器码、配置文件语义、已有页面文本和用户操作顺序保持兼容。
4. 每一迁移批次均能独立运行现有自动化、静态检查和浏览器主路径 Gate；禁止“先大搬家、最后集中修回归”。

### 0.2 非目标（硬边界）

| 非目标 | 明确说明 |
|---|---|
| 新增产品功能 | 不新增连接组类型、路由算法、聚合策略、诊断能力、页面入口或配置字段 |
| 修改 API 契约 | 不新增/删除/改名对外 API；不改 HTTP 方法、核心字段、错误 `type/code` 或成功/失败语义 |
| 调整配置数据模型 | 不迁移配置格式、不改字段含义、不重置本地配置；内部 DTO/适配器不得改变落盘 JSON 语义 |
| 视觉或交互改版 | 不调整页面布局、文案、按钮层级、字段顺序、默认值、自动保存时机或操作步骤 |
| 顺带修复/优化 | 发现问题仅登记为后续事项；除非属于重构直接引入的 P0 回归，不将行为修正混入本版本 |
| 一次性框架替换 | 不切换 Web 框架、不改前端框架、不引入大规模状态管理库或构建体系重写 |
| 性能指标承诺 | 本期以“无明显退化”为门槛；性能专项优化另立需求和基准 |

> 判定原则：若用户、客户端、现有自动化或配置文件能够观察到差异，则默认不属于 v0.5.6；必须单独立项并得到产品确认后才可纳入。

## 1. 背景、问题与证据

### 1.1 当前代码职责地图（2026-07-11 基线）

| 区域 | 实际规模/职责 | 主要耦合风险 |
|---|---|---|
| `app.py` | 约 5,702 行；数据对象、`ConfigStore`、日志、上游请求、候选选择、健康状态、流式转发、HTTP 路由、静态文件、启动均在同一文件 | 任意领域改动都可能误触 HTTP、运行时或持久化逻辑；测试和 code review 很难聚焦 |
| `ConfigStore` | `app.py:446-957`；配置加载/保存、连接组/模型/聚合模型与成员 CRUD、校验、冷却刷新 | 配置规则与 HTTP 层、运行时层同文件，领域规则难复用/单测 |
| `ArkProxyRouter` | `app.py:976-3615`；日志、请求诊断、上游候选、WAF、健康/冷却、`call/stream` 转发 | 高复杂运行态与日志/诊断/协议处理集中，改一处难判断是否影响请求主链 |
| `RouterHandler` | `app.py:3622-5637`；静态资源、JSON/文件处理、配置/模型/聚合/日志/测试/代理 API 分发 | 端点分支巨大，HTTP 参数解析、领域调用、响应组装交织 |
| 启动逻辑 | `app.py:5640-5698`；初始配置、端口、服务创建、main | 应成为稳定装配入口，当前与全部实现同文件共存 |
| `static/js/config-tab.js` | 约 1,946 行；连接组/模型/聚合渲染、表单草稿、校验、事件绑定、自动保存、运行态刷新、批量导入、收益看板 | 表单编辑与运行态刷新容易互相覆盖；DOM、状态和 API 副作用耦合，局部修改难回归 |
| 配置页运行态刷新 | `config-tab.js` 包含 `onRuntimeStateUpdate`、状态局部 patch、冷却计时 | 若误用整页 render，会损失焦点、脏表单、展开态；v0.5.5 已明确要求 dirty guard + diff patch，必须原样保留 |

### 1.2 已知必须保持的产品契约

| 契约 | 当前依据 | v0.5.6 要求 |
|---|---|---|
| 连接组/模型/聚合模型对象 | `ConnectionGroup`、`ModelConfig`、`AggregateModel`、`AggregateMember` | 对象含义、ID、排序、启停、路由 Key、客户端别名与配置序列化语义不变 |
| 配置持久化 | `ConfigStore.load/save` 与现有本地配置 | 旧配置可直接启动；保存后字段语义与兼容性不变；不得要求用户重新配置 |
| 代理请求 | `ArkProxyRouter.call/stream` | 非流式/流式、候选选择、WAF 兼容、超时、冷却、聚合 fallback 与日志归并行为不变 |
| HTTP API | `RouterHandler.do_GET/do_POST/do_PUT/do_DELETE` | 已有 `/api/*`、`/v1/*` 路径、方法、参数、响应 shape、状态码与中文 `message` 保持不变 |
| 配置页编辑体验 | v0.5.5 PM Gate | 新建 relay 默认值、切 provider 清默认地址、草稿状态、补全字段聚焦、显式保存、dirty guard、局部状态刷新均保持不变 |
| 客户端/运维体验 | Dashboard、日志、测试、导入导出 | 入口、按钮文案、操作顺序与可见结果不改变；不可因模块迁移出现整页闪烁或焦点丢失 |

### 1.3 问题陈述

> 在不改变产品行为的前提下，如何将高耦合的后端单文件和配置页单文件拆成可理解、可测试、可分批迁移的模块，使后续修改能定位到明确领域，并且每一步都有真实回归证据？

### 1.4 关键假设与验证方式

| 假设 | 风险 | 本期验证方式 |
|---|---|---|
| 现有自动化可覆盖主要服务行为 | 仅跑测试不足以发现 UI 脏表单/流式回归 | 补足“契约快照 + 浏览器主路径”双 Gate，不以单一 pytest 结果下结论 |
| 模块迁移可保持导入关系稳定 | 循环依赖可能迫使临时全局变量或隐式耦合 | 每批次输出依赖图并执行启动 smoke；禁止新增跨层反向 import |
| 配置页可用局部 patch 保持编辑状态 | 将运行态刷新从页面协调层拆出后可能漏更新 | 针对输入中、下拉展开、错误提示、冷却倒计时做人工回归 |
| 抽取纯领域逻辑不会改变异常语义 | 旧逻辑可能隐含依赖 HTTP/全局状态 | 将现有成功/失败响应、日志事件、配置读写样本作为迁移前后对照 |

## 3. 目标架构边界（产品冻结口径）

> 本节定义“职责应属于哪里”，不强制某个实现框架、文件名或类名。架构师/研发可在不突破边界的前提下决定最终目录与内部命名；若变更对外契约，需回到产品评审。

### 3.1 后端目标边界

```text
应用入口/装配层
  └─ 创建 Store、运行时服务、HTTP Server；注册路由；处理启动参数

HTTP 适配层
  └─ 请求解析、鉴权上下文提取、调用应用服务、响应/错误序列化

领域与应用服务层
  ├─ 配置领域：连接组、模型、聚合模型、成员的校验与 CRUD
  ├─ 路由运行时：候选选择、上游转发、流式生命周期、冷却/恢复
  ├─ 观测诊断：日志、运行态、诊断、导出
  └─ 上游适配：请求头、认证、WAF 兼容、模型列表获取

基础设施层
  └─ 配置文件读写、日志文件读写、HTTP Client、时间/ID 等外部能力
```

| 边界 | 允许职责 | 明确禁止 |
|---|---|---|
| 应用入口/装配层 | 依赖实例化、Server 创建、启动/停止 | 写业务判断、拼装具体 API 响应、直接处理配置字段 |
| HTTP 适配层 | 解析 request、调用服务、按既有契约返回 response | 直接改配置文件、实现候选算法、把业务状态藏在 Handler 实例 |
| 配置领域 | 配置校验、实体 CRUD、领域约束、持久化调用协调 | 读取 HTTP request、输出 HTML/HTTP response、执行真实上游请求 |
| 路由运行时 | 请求转发、候选与健康状态、流式生命周期 | 解析管理页表单、直接依赖具体静态资源路径 |
| 观测诊断 | 日志读写、诊断计算、运行态快照、导出数据 | 改变实际路由结果或绕过既有脱敏规则 |
| 上游适配 | 上游 URL/认证/header/兼容策略/模型列表调用 | 决定本地产品对象的 CRUD 或页面显示文案 |

### 3.2 后端建议迁移单位（顺序可调整，边界不可倒置）

| 批次 | 建议抽取内容 | 迁移后的职责收益 | 本批禁止变化 |
|---|---|---|---|
| B1 | 数据对象、配置序列化/校验、`ConfigStore` | 先形成可独立测试的配置领域 | 配置 JSON 字段、校验文案、CRUD 行为、导入导出语义 |
| B2 | 日志文件读写、运行态/诊断查询与导出 | 观测能力脱离转发主链，提升定位性 | 日志格式、脱敏、事件名、筛选/导出结果 |
| B3 | 上游请求头/认证/WAF/模型列表适配 | 上游兼容策略集中，避免散落调用 | provider 规则、请求头、超时、错误分类 |
| B4 | 路由运行时（候选、冷却、call/stream） | 明确请求生命周期与健康状态所属 | 路由顺序、fallback、流式输出、冷却/恢复 |
| B5 | HTTP 路由分组与应用入口收口 | Handler 仅保留适配职责，`app.py` 成为装配入口 | 所有已有路径、方法、响应字段与状态码 |

**依赖原则：** HTTP 层可依赖应用/领域服务；应用服务可依赖基础设施接口；领域层不得反向 import HTTP Handler 或页面实现。B1-B4 的业务模块不得反向引用 B5 的路由模块。

### 3.3 前端目标边界

```text
配置页协调入口（Tab Controller）
  ├─ 渲染区域/页面装配
  ├─ 连接组表单领域：草稿读取、默认值、字段可见性、校验、错误呈现
  ├─ 模型/聚合表单领域：各自表单状态与校验
  ├─ 配置动作：保存、删除、克隆、获取模型、导入导出、恢复等 API 编排
  ├─ 运行态视图：runtime-state、冷却计时、状态局部 patch
  └─ 通用 UI：事件委托、下拉选择、复制、弹层、Toast（复用现有能力）
```

| 边界 | 允许职责 | 明确禁止 |
|---|---|---|
| 配置页协调入口 | 初始化、选择当前子视图、协调模块通信、销毁清理 | 承载大段字段校验、直接拼接所有业务 HTML、包含具体 API 业务细节 |
| 表单领域模块 | 读/写草稿、默认值、provider 切换规则、字段校验、错误/焦点 | 轮询运行态、整页 render、直接替代通用 API 错误处理 |
| 配置动作模块 | 调用既有 API、提交前读取表单、成功后按既有规则刷新/跳转 | 擅自改 API payload、改变保存/测试时机、持有长期 DOM 状态 |
| 运行态视图模块 | runtime-state 消费、行级 diff patch、冷却倒计时、可见状态同步 | 覆盖脏表单、重建整个 config panel、触发表单自动保存 |
| 纯渲染/组件模块 | 接收明确数据并输出局部 HTML/DOM 更新 | 读取全局隐式状态、发起网络请求、决定业务操作顺序 |

### 3.4 前端重点保护规则

| ID | 规则 | 验收要求 |
|---|---|---|
| FE-R01 | 编辑保护 | 输入 Base URL/API Key/模型字段时，运行态刷新不得覆盖输入值、光标、焦点、字段错误、下拉展开或未保存状态 |
| FE-R02 | 局部更新 | 运行态轮询仅 patch 状态单元/倒计时；无数据变化时不操作 DOM；不得用 `panel.innerHTML` 重建配置页 |
| FE-R03 | 草稿单一来源 | 新建连接组与编辑表单的当前草稿必须从当前表单或明确草稿状态派生；“补全字段”校验的对象必须是当前草稿 |
| FE-R04 | 默认值边界 | 新建 relay 的默认 Base URL、切换 provider 时仅清理系统默认 URL、用户手填 URL 不被清除，行为与 v0.5.5 一致 |
| FE-R05 | 副作用边界 | 保存、获取模型、恢复、删除、导入等网络副作用仅由动作模块触发；render/状态推导函数不得隐式发请求 |
| FE-R06 | 生命周期清理 | Tab 切换、页面销毁后，计时器、全局鼠标/键盘事件与轮询订阅不得重复绑定或继续修改不可见 DOM |

## 4. 迁移策略与范围拆解

### 4.1 总体原则：绞杀式小步迁移，不做并行双逻辑

```text
冻结现有行为基线
  ↓
抽取一块内聚职责（旧入口改为调用新模块）
  ↓
执行该块契约测试 + 全量回归 + 人工关键路径
  ↓
确认无行为差异后，再移下一块
```

- 旧入口在迁移期间仅作为兼容调用点，不能长期保留两套独立业务实现。
- 禁止先复制逻辑、两边同时修改、最后再判断谁为准；每批完成后应只有一个实现来源。
- 允许为降低迁移风险建立薄 Adapter，但 Adapter 只能做参数映射/依赖注入，不得复制业务判断。
- 任一批出现用户行为/API/配置不一致，停止向下一批扩展，先恢复或修正当批。

**目标：** 将 `ArkProxyRouter` 内的候选选择、健康/冷却状态、普通请求与流式请求生命周期抽取为显式运行时服务；`ArkProxyRouter` 保留兼容 facade，对 `RouterHandler`、HTTP API、现有配置对象和调用方保持原有接口与行为。

#### A. 状态、锁与请求生命周期

1. **共享状态：** 候选可用性、失败记录、冷却截止时间、恢复状态及其既有锁由 `health` 统一拥有；调用方仅能读取快照或提交既有结果事件，不得直接改写内部状态。
2. **锁语义冻结：** 必须保留迁移前锁的获取顺序、覆盖范围、释放时机和异常路径；不得调整粒度、重入语义，或改变网络 I/O、流式 yield 所处的锁范围。
3. **生命周期：** `call` 负责非流式从候选快照到日志完成的原有闭环；`stream` 负责流式从候选快照、首包、chunk、usage 到结束/异常日志收口的原有闭环；`health` 不自行发请求、不决定重试、不写 HTTP 响应。
4. **首包后不切换：** 一旦按既有规则向客户端产生首个有效上游输出，后续异常仅能走原流式失败/结束路径；不得回到候选选择或触发 fallback。
5. **日志：** M2a 观测模块继续提供既有记录能力；M3 仅在原有时点调用，不得提前、延后、合并、拆分、重复“尝试、首包、成功、失败、流结束”等记录。

#### B. 显式依赖注入与兼容规则

```text
app.py 装配层
 ├─ ConfigStore / 配置只读 Port
 ├─ UpstreamAdapter（M2b）
 ├─ Observability / DebugCapture（M2a）
 ├─ Clock、既有锁与运行态容器
 └─ RouterRuntimeService(dependencies)
      ├─ CandidateSelector
      ├─ HealthManager
      ├─ call 生命周期
      └─ stream 生命周期
ArkProxyRouter（兼容 facade） → RouterRuntimeService
RouterHandler → ArkProxyRouter（不感知 M3 内部）
```

- 依赖必须由现有装配位置显式创建并注入；不得新增模块级 Router、Store、Adapter 或 Lock 单例。
- `RouterRuntimeService` 仅依赖配置读取、M2b `UpstreamAdapter`、M2a 观测、时间/状态 Port；不得反向 import `RouterHandler`、HTTP request/response、静态资源或管理 API。
- M3 不修改 M2b Adapter 的 URL、认证、header、WAF、模型获取和内部错误上下文；也不修改 `RouterHandler` 的参数解析、HTTP 状态码、`type/code/message` 映射。
- facade 只做参数转交、依赖持有和返回值/异常原样传递。同一请求路径不得同时执行旧、新两套路由业务实现；内部 DTO 不足时仅新增私有纯数据上下文，不得传入 Router、Handler 或 HTTP 对象整体。

## 5. 行为一致性验收标准

### 5.0 当前实施任务：M5+M6 前端配置页职责收口

**问题与目标：** `static/js/config-tab.js` 当前同时负责整页 render、表单草稿/校验、保存/导入等动作和 runtime-state 更新。该批只调整职责归属，目标是让 `ConfigTab` 成为页面协调入口，且不改变任何可见 UI、API payload、默认值、保存时机或交互顺序。

| 子域 | 允许迁移 | 冻结边界 |
|---|---|---|
| 表单草稿与校验 | 新建/编辑连接组、模型、聚合对象的当前草稿、provider 切换、字段可见性、补全/校验/焦点规则 | relay 默认 Base URL、用户手填 URL 保护、字段顺序、错误文案、显式保存时机不变 |
| 配置动作编排 | 保存、删除、克隆、获取模型、导入导出、恢复、排序等既有 API 调用和结果后的既有刷新 | API 路径、method、payload、成功/失败提示、确认/预览顺序不变 |
| 运行态视图 | `runtime-state` 消费、状态单元 diff patch、冷却倒计时和 tab dispose | 不得重建编辑中的 config panel；不得覆盖 dirty 表单、焦点、光标、下拉展开或错误提示 |

**实施硬规则：**

1. 先盘点 `config-tab.js` 当前 `render`、表单读取/校验、动作、`onRuntimeStateUpdate`、timer/event listener 的调用关系，再抽取；不允许先复制为第二套逻辑。
2. `panel.innerHTML` 仅可保留在明确切换选中对象或首次进入所需的全量 render 路径；轮询/运行态刷新禁止调用全量 `render()` 或重建 panel。
3. 每个模块的输入、输出和 `dispose` 必须显式；不得通过新的全局可变状态、隐式 DOM 查询或重复 document listener 传递业务状态。
4. 新建草稿与编辑草稿保持单一真源；保存/测试仅从当前草稿读取，不能回退读取旧 Store 造成校验或 payload 偏差。
5. 该阶段为单一独立提交；实现前后均须检查 `git diff --check`、敏感/运行产物，并保留可整批回滚点。

**进入 Gate：** 基线 `b267cfe` 工作区干净；先新增/调整前端专项契约，覆盖选中对象切换、dirty 表单、provider 切换、动作 payload、runtime diff patch、timer/listener dispose。

### 5.3 发布阻断条件

出现下列任一项，v0.5.6 不得发布：

- API 路径、HTTP 方法、响应字段、机器码或中文 message 出现未批准变化；
- 已有配置无法读取、保存后字段丢失/变形，或需要用户重新填写；
- 普通/流式请求、fallback、WAF、冷却恢复、日志生命周期出现行为差异；
- 配置页出现整页轮询闪烁、编辑中字段被覆盖、焦点丢失、重复事件/计时器；
- v0.5.5 已验收的 relay 默认值、provider 切换、草稿校验、显式保存任一项回归；
- 自动化、静态检查、契约对照或人工 Gate 未完成/失败；
- 为通过重构临时引入真实 Key、日志数据库、备份文件等敏感/运行时产物进入提交集。

### 6.2 依赖关系

```text
M0 基线
```

- M1 是后端迁移的优先前置，避免 HTTP/运行时模块各自持有配置规则。

## 11. 结论

v0.5.6 是一次**边界先于代码形态**的重构：不以“拆成更多文件”作为完成，而以现有用户、客户端和配置在迁移前后**没有可观察差异**为交付标准。

实施必须遵守：先固化基线、逐域迁移、逐批回归、失败即止损；任何新能力、体验优化、接口演进都不应搭载本版本进入发布范围。
