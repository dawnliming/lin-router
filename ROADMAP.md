# Lin Router 优化升级 PRD

## 1. 背景
Lin Router 目前已经具备本地 OpenAI 兼容路由、Hermes 接入、Codex++ 接入、中转站代理、自动模型切换、冷却机制和请求日志等能力。

在实际使用中，当前版本仍存在几个典型问题：
- 连接组和模型的管理方式还不够直观，页面上信息密度和操作方式不够统一。
- 中转站、火山方舟、通用 OpenAI 代理三类模式的边界还不够清晰，容易互相干扰。
- 自动调度已经存在，但在模型失效、429、冷却、恢复等场景下还需要更稳定的策略。
- 请求日志虽然有，但排障深度还不够，遇到“为什么这个时间还有请求”“为什么没命中缓存”这类问题时，定位成本较高。
- 前端配置页面有些内容重复、分散，用户在新增连接组和模型时容易产生误操作。

这份文档的目标，就是把当前“能跑”的状态，收敛成“能长期稳定使用、能看懂、能调试、能扩展”的状态。

## 2. 核心目标

### 2.1 产品目标
- 让不同接入方式彼此隔离，但共用同一套本地路由入口。
- 让用户在页面上能清楚知道：当前连接组是什么、当前模型是什么、为什么会切换、为什么会冷却。
- 让 Hermes、Codex++、普通代理都能稳定接入，不因某个模式的特殊逻辑影响其它模式。
- 让中转站场景下的自动切换真正可用，尽量减少人工干预。

### 2.2 体验目标
- 配置更少、更清楚。
- 页面更整洁，连接组与模型关系一目了然。
- 重要状态更显眼：可用、冷却、失效、WAF、自动模型。
- 排障信息更完整，不需要频繁翻代码或猜请求内容。

### 2.3 工程目标
- 保持 OpenAI 兼容接口稳定。
- 中转站增强逻辑尽量只影响 relay 模式。
- 避免为了某一个接入方修改后，破坏另一个接入方的请求语义。
- 后续新增能力时，尽量围绕连接组扩展，而不是把逻辑散落到多个地方。

## 3. 现阶段范围

### 3.1 连接组管理
连接组是路由与隔离的基本单元。后续所有配置、状态、调度都应尽量围绕连接组组织。

需要支持：
- 连接组名称
- 模式类型：火山方舟 / 中转站 / 通用 OpenAI 代理
- base_url
- 组级 key 或组内模型 key，按模式区分
- 是否启用自动调度
- 是否启用 WAF 兼容
- 冷却时间
- 组内模型列表
- 连接组复制
- 连接组编辑与删除

### 3.2 模型管理
模型是连接组内的具体上游目标。

需要支持：
- 模型名称
- 上游模型标识
- EP ID 或价格组标识
- 上游 key
- 可用状态
- 最近成功时间
- 最近错误信息
- 冷却截止时间
- 冷却原因
- 手动启用 / 禁用
- 自动获取上游模型

### 3.3 自动调度
自动调度主要用于中转站场景，目标是当某个模型失效时能够快速切到下一个可用模型。

需要支持：
- 同组模型按优先级或顺序轮询
- 遇到 429、5xx、网络错误时自动尝试下一个模型
- 失败模型进入冷却
- 冷却到期后自动恢复可尝试
- 支持手动立即启用，且启用后清除冷却状态
- 冷却时间可配置，且跟随连接组

### 3.4 Hermes / Codex++ / 通用代理边界
- Hermes 继续走严格 key 识别连接组。
- Codex++ 保持原始兼容行为，不被中转站特殊逻辑污染。
- 通用代理场景应尽量原封不动转发请求。
- WAF 兼容只在中转站相关连接组中出现。
- 自动获取模型按钮只在需要的模式中显示。

### 3.5 日志与排障
需要把请求日志从“能看一眼”升级到“能定位问题”。

至少要记录：
- 请求时间
- 请求路径
- 命中的连接组 / 模型
- 状态码
- 耗时
- prompt tokens
- completion tokens
- total tokens
- cached tokens
- 请求详情摘要
- 是否为自动切换、重试或冷却触发

希望后续支持：
- 持久化保存
- CSV 导出
- 按时间范围筛选
- 按连接组筛选
- 按状态筛选

## 4. 详细需求

### 4.1 连接组页面
#### 需求
- 连接组列表要更适合管理。
- 连接组信息应当作为主视图，模型作为展开项或附属项。
- 编辑、复制、删除等操作应放在连接组本身上，而不是分散在模型层。
- 当连接组较多时，页面不能显得太拥挤。

#### 预期表现
- 列表可以使用折叠面板或表格 + 展开行。
- 折叠后能看到核心信息：组名、模式、base_url、状态、自动调度、WAF、模型数量。
- 展开后能看到组内模型的详细配置。

### 4.2 模型配置
#### 需求
- 相同 base_url 下尽量复用同样的 key 配置，减少重复输入。
- 中转站场景下，一个 key 对应一个分组，分组内可以有多个模型。
- “自动获取模型”按钮触发后，不应该直接把整组模型铺到连接组顶层，而应该填充到上游模型下拉框。
- 下拉框默认应为空，只有点击获取后才有内容。
- 保存模型后应重置表单，并清空上次获取的下拉项，避免残留影响下一次操作。

#### 预期表现
- 模型行内可选择上游模型。
- 若未点击获取，上游模型选择框不应误导用户。
- 同组多个模型的管理逻辑清晰。

### 4.3 自动切换与冷却
#### 需求
- 仅中转站模式需要自动冷却与自动切换。
- 失败时按顺序尝试下一模型。
- 冷却时间可配置，且是组级配置。
- 冷却中的模型需要明显状态标识。
- 手动“启用”应立刻生效，并清除冷却时间。
- 默认加载时，非中转站组不展示冷却相关配置。

#### 预期表现
- 失败后切换自然，不需要用户干预。
- 冷却机制不会误伤火山方舟或通用代理组。

### 4.4 请求与缓存语义
#### 需求
- 接入 Hermes 时，保留其请求语义，不要随意改写影响上游识别。
- 接入 Codex++ 时，尽量保持它原生请求结构。
- 中转站兼容逻辑要尽量薄，避免影响缓存命中率。
- 仅当确实需要兼容 WAF 或中转站规则时，才做额外处理。

#### 预期表现
- 不同客户端请求都能稳定路由。
- 缓存命中率不应因为路由层额外改写而显著下降。

### 4.5 日志系统
#### 需求
- 支持查看每一次请求的详细信息。
- 能解释“为什么后面还有请求”“是谁触发了自动切换”“哪个模型进入了冷却”。
- 支持导出，方便在外部排查。

#### 预期表现
- 日志列表足够完整，能直接帮助定位问题。
- 后续可扩展到文件持久化。

## 5. 视觉与布局方向
- 页面以管理效率为主，不做花哨装饰。
- 连接组信息优先，模型信息折叠展示。
- 操作按钮大小统一，避免按钮高度和行高不协调。
- EP 宽度适当收窄，操作列适当放宽，保证一行内能看清常用操作。
- 仅在需要的模式里显示相关能力，减少噪音。

## 6. 非目标
以下内容不作为当前阶段重点：
- 大规模重构请求协议。
- 改造为云端 SaaS。
- 引入复杂账号体系。
- 做成多用户协作后台。
- 把所有模式强行统一成一个配置结构。

## 7. 迭代顺序建议
### 第一阶段
- 日志持久化或更强的查询能力。
- 连接组列表与模型列表结构重排。
- 冷却状态和手动启用逻辑修正。

### 第二阶段
- 中转站自动获取模型与下拉选择流程整理。
- Hermes / Codex++ / relay 的请求边界清理。
- 自动调度策略稳定化。

### 第三阶段
- README 和发布文档同步。
- 配置导入导出整理。
- 打包、预览、桌面端发布流程优化。

## 8. 验收标准
- 用户能清楚区分每个连接组的用途。
- 中转站场景下，模型失效后能自动切换到其他模型。
- 冷却中的模型可手动立即启用。
- Hermes / Codex++ / 通用代理互不干扰。
- 日志能帮助解释“为什么某时间还在请求”。
- 前端配置流程更顺，不再频繁出现无意义的重复填写。

## 9. 后续维护原则
- 新功能优先挂在连接组或组内模型上。
- 能只影响中转站的逻辑，不要影响其它模式。
- 能复用现有结构的，不重造一套。
- 每次修改后，都要能从日志和 UI 上看出变化。
